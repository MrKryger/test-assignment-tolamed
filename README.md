### Списание бонусов (`POST /users/:id/spend`)

- **Идемпотентность** по ключу `(user_id, requestId)`
- Поддержка `Idempotency-Key` в заголовке (приоритет) или `requestId` в теле
- Защита от дублей и конфликтов (409 если тот же requestId, но другая сумма)
- Баланс не может уйти в минус при гонках
- Просроченные начисления (`expires_at < NOW()`) не учитываются
- Ответы:
    - Первый успешный запрос → `{ success: true, duplicated: false }`
    - Повтор того же запроса → `{ success: true, duplicated: true }`
    - Конфликт payload → `409`
    - Недостаточно средств / плохой запрос → `400`

### Автоматическое списание просрочки (`POST /jobs/expire-accruals`)

- Задача ставится с фиксированным `jobId: "expire-accruals"` → не дублируется
- Worker находит все `accrual` с `expires_at < NOW()`
- Для каждого создаёт `spend` с `request_id = "expire:<accrual_id>"`
- Если такая трата уже существует → пропускает (идемпотентность)
- Retry: 3 попытки с экспоненциальным backoff (1с → 2с → 4с)
- Повторный запуск задачи ничего не ломает

## Как обеспечена надёжность

| Уровень          | Что защищает                              | Как реализовано                                      |
|------------------|--------------------------------------------|-------------------------------------------------------|
| База данных      | Дубли по requestId                        | Уникальный частичный индекс `(user_id, request_id)`  |
| База данных      | Отрицательные / нулевые суммы             | CHECK constraint `amount > 0`                        |
| Код + БД         | Гонка при параллельных списаниях          | `SELECT ... FOR UPDATE` + транзакция                 |
| Код              | Повтор запроса с тем же requestId         | Проверка существования + возврат duplicated: true    |
| Код              | Просроченные бонусы в балансе             | Фильтр `expires_at IS NULL OR expires_at >= NOW()`   |
| Очередь          | Дубли задач                               | Фиксированный jobId                                  |
| Очередь          | Временные сбои                            | 3 попытки + backoff                                  |
| Очередь + код    | Повторная обработка просрочки             | Идемпотентные requestId вида `expire:<id>`           |

## Основные компромиссы (сделано за 4–6 часов)

- Используется `FOR UPDATE` на все строки пользователя (а не advisory lock или материализованный баланс)
- Баланс пересчитывается каждый раз по всем транзакциям (нет кэша / отдельной таблицы)
- Просроченные начисления обрабатываются по одному в цикле (не батчами)
- Нет явной проверки существования пользователя перед списанием (падает в «не хватает бонусов»)
- Тесты используют общую БД + генерацию уникальных пользователей

Для большинства реальных объёмов данных и нагрузки этих решений достаточно.  
При росте до миллионов транзакций на пользователя стоит добавить:

- материализованный баланс в таблице `user_balances`
- advisory locks вместо FOR UPDATE
- батчинг в expire-worker
- отдельную проверку пользователя
